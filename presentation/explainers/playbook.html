01

Context Engineering

How you frame the problem determines whether the agent can solve it.

Agents are only as good as the context they receive. Context engineering is the discipline of preparing documents, prompts, and acceptance criteria so that an agent can work reliably without constant hand-holding.

Chunking Documents

  • Break large documents into semantic sections, not fixed-length blocks
  • Preserve headings, section numbers, and cross-references as metadata
  • Include surrounding context (e.g., the parent section title) when passing a chunk
  • Use overlap between chunks to avoid losing information at boundaries

Prompt Framing

  • State the goal, the constraints, and the expected output format up front
  • Provide examples of correct and incorrect output (few-shot patterns)
  • Specify what the agent should do when uncertain: ask, flag, or skip
  • Keep prompts modular so they can be composed into larger workflows

Designing Checklists

  • Write acceptance criteria as boolean assertions the agent can evaluate
  • Each checklist item should be independently verifiable
  • Include negative checks: "this must NOT contain X"
  • Checklists become your automated review gates and regression tests
02

Orchestration Patterns

Match the orchestration style to the complexity of the task.

Not every problem needs the same kind of agent coordination. Use Cynefin domains and OODA loops to decide which pattern fits.

Conductor

One orchestrator delegates tasks to specialist agents and aggregates results.

Use when: tasks are well-defined and can be decomposed in advance (Cynefin: Clear)

Spec-Driven

Agents work from a shared specification, validating their output against acceptance criteria.

Use when: the problem is complicated but analyzable with expertise (Cynefin: Complicated)

Swarm

Multiple agents explore in parallel, with adversarial review and convergence checks.

Use when: the solution space is large and discovery is needed (Cynefin: Complex)

OODA Loop

Observe-Orient-Decide-Act cycle with human checkpoints at each stage.

Use when: stakes are high or outcomes are uncertain. Always start here.

The OODA loop is not just a decision framework. It is your trust-building protocol: observe what the agent does, orient on whether it makes sense, decide whether to proceed, and only then let it act. -- Playbook principle
03

Skills & Compounding

Every solved problem should become a reusable skill.

A skill is a codified piece of problem-solving: a prompt template, a checklist, a tool invocation pattern. Skills compound over time because each new skill can be composed with existing ones.

Building a Skill

  • Identify a task you perform repeatedly with similar steps
  • Write the prompt, the expected input format, and the acceptance criteria
  • Test it on three representative examples before committing
  • Store skills in version control with your codebase

Composing Skills

  • Chain skills: output of one becomes input to the next
  • Gate skills: one skill validates the output of another
  • Fork skills: run multiple skills in parallel, merge the best result
  • Skills are your institutional memory for agent workflows

Compounding Effect

  • Week 1: one skill saves 30 minutes per use
  • Week 4: five skills chain together for an end-to-end workflow
  • Week 12: skills become the team's standard operating procedures
  • The brownian ratchet: each skill locks in a small gain permanently
04

Safety & Containment

Trust is earned incrementally. Containment is how you earn it.

Before an agent touches production data or customer-facing systems, it must operate within containment boundaries that limit blast radius and create audit trails.

Sandboxing

  • Run agents in isolated environments: containers, worktrees, or staging
  • Use separate branches for agent work; never let agents push to main
  • Limit filesystem and network access to only what the task requires
  • Treat agent output as untrusted input until reviewed

Review Gates

  • Every agent output goes through a review step before merging or deploying
  • Use pull requests as natural review gates with human approval
  • Adversarial review: a second agent or human specifically looks for errors
  • Automate what you can (linting, tests), but keep human judgment in the loop

Audit Trails

  • Log every agent action: what was requested, what was produced, who approved
  • Use git history as a natural audit log for code and document changes
  • Structured output (JSON, YAML) makes agent decisions machine-verifiable
  • Retention policy: keep agent interaction logs for compliance review
05

The Brownian Ratchet

Underpromise, deliver, iterate. Progress through validated increments.

Agents will not produce perfect output on the first try. The brownian ratchet principle says: accept imperfect output, validate it, lock in the gains, and iterate. Eventual consistency beats initial perfection.

Iterative Validation

  • Run the agent, review the output, fix what's wrong, run again
  • Each iteration should reduce the error surface, not just shuffle errors
  • Set a convergence criterion: "stop when the diff between iterations is below threshold"
  • Track metrics across iterations to prove progress

Eventual Consistency

  • Testing validity and impact matters more than initial correctness
  • Small, verified steps beat large, unverified leaps
  • Use checkpoints: save intermediate state so you can resume, not restart
  • The ratchet only moves forward -- never discard validated work

Quick Start Checklist

Print this. Pin it up. Start checking things off.

Monday Morning Checklist

  1. Identify your most painful repetitive workflow
  2. Write a one-paragraph description of what "done" looks like for that workflow
  3. Build a single skill: prompt + acceptance criteria + example input/output
  4. Run the skill in a sandboxed environment on real (non-production) data
  5. Review the output manually. Note what's right, what's wrong, what's missing
  6. Iterate: fix the skill based on review, run again, compare to previous output
  7. Set up a review gate: PR-based workflow or human sign-off step
  8. Use Cynefin to classify three more workflows. Only automate Clear and Complicated
  9. Compose two skills into a chain. Validate the handoff between them
  10. Share your playbook with your team. Start building institutional muscle memory
